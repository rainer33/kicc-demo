# 결제 통합 소스 분석 및 추진 전략

## 1) 배경
현재 결제 환경은 다음과 같습니다.
- `LightPay`: 기존 운영중인 결제 모듈
- `KICC`: 신규 도입 결제 모듈
- 신규 채널 도입: `키오스크`, `POS`, `오프라인 결제`

핵심 난점은 결제 API 호출 자체보다, 결제 직전 금액/할인/옵션/세트/세금이 얽힌 **상품 구성 로직의 복잡성**입니다.

## 2) 목표
1. 결제 모듈(LightPay/KICC) 전환 및 병행 운영이 가능한 구조 확보
2. 상품 복잡도와 결제 처리 책임을 분리하여 결제만 독립 테스트 가능하게 구성
3. 키오스크/POS/오프라인 채널을 단일 결제 코어로 흡수
4. 운영중 시스템 무중단 전환(점진적 라우팅 + 롤백 가능)

## 3) 분석 범위(소스 분석 우선순위)
### A. 결제 진입점 전수 파악
- 웹/앱/키오스크/POS/관리자/배치에서 결제를 시작하는 API 및 서비스 목록화
- 승인/취소/부분환불/전액환불/실패 재처리 경로 식별

### B. 결제-상품 결합 지점 탐색
- 결제 서비스 내부에 상품 계산/가격 보정/쿠폰/프로모션/세금 계산 로직이 섞여 있는지 확인
- 결제 시점에 DB에서 재계산하는 로직과 사전 계산값을 사용하는 로직 구분

### C. 상태머신 및 정합성
- 주문/결제/환불 상태 전이표 작성
- 멱등키, 중복 승인 방지, 콜백 재전송/지연 처리 정책 확인
- 오프라인 결제 동기화 시 상태 정합성 규칙 정의

### D. 외부 연동 계층 구조
- LightPay/KICC 연동 코드를 공통 흐름과 PG 전용 구현으로 분리 가능한지 평가
- 서명 검증, 거래번호 매핑, 에러코드 매핑 일관성 점검

### E. 운영 리스크 포인트
- 장애 시나리오(타임아웃/네트워크 단절/콜백 누락/중복)
- 수동 보정 경로와 자동 보정 배치 존재 여부
- 감사 로그/이력 추적 가능성

## 4) 권장 아키텍처
### A. 결제 코어와 PG 어댑터 분리
- `PaymentOrchestrator` (공통 흐름)
- `PaymentProviderAdapter` (LightPayAdapter, KiccAdapter)

### B. 상품 로직과 결제 로직 절단
- 결제 입력을 상품 원본이 아닌 `OrderSnapshot(최종 금액/세금/할인/라인요약)`으로 고정
- 상품 계산은 별도 도메인, 결제는 스냅샷 기반으로만 처리

### C. 채널 독립 설계
- 채널(Web/App/Kiosk/POS)은 UI/단말 이벤트만 다르게 처리
- 승인/취소/환불/멱등/콜백 처리 코어는 공통 사용

### D. 오프라인 결제 대응
- 승인 이벤트 수신 시점과 서버 동기화 시점을 분리
- 동기화 지연/중복 전송 대비 멱등 처리 필수

## 5) 단계별 추진 전략
### 1단계: AS-IS 맵 작성
- 결제 관련 컨트롤러/서비스/리포지토리/배치 전수 목록
- 상태 전이표/시퀀스 다이어그램/테이블 관계 문서화

### 2단계: 공통 결제 계층 도입
- LightPay 경로를 먼저 Adapter로 감싸 기존 동작 동일성 확보
- 기존 운영 흐름은 그대로 유지(기능 플래그 Off)

### 3단계: KICC 병행 도입
- 플래그/가맹점/채널 기준 라우팅으로 점진 전환
- 동일 주문 유형에 대해 A/B 또는 단계적 트래픽 전환

### 4단계: 키오스크/POS/오프라인 통합
- 단말 이벤트를 표준 이벤트 모델로 수집
- 결제 코어로 승인/실패/취소/환불 일관 처리

### 5단계: 안정화/보정 체계 완성
- 실패 재처리, 정합성 보정 배치, 감사 로그 강화
- 운영 대시보드(상태 분포, 실패율, 재처리율) 연결

## 6) 테스트 전략
### A. 결제 독립 테스트
- 상품 계산을 제외하고 `OrderSnapshot fixture`만으로 승인/취소/환불 테스트
- PG Mock 응답으로 성공/실패/타임아웃/지연 콜백 재현

### B. 상태머신 테스트
- `READY -> APPROVED -> CANCELED/REFUNDED` 전이 검증
- 재전송 콜백/역순 콜백/중복 승인 방어 검증

### C. 동시성/멱등 테스트
- 동일 키 동시 요청
- 단말 이벤트 중복 수신
- 네트워크 재시도 폭주 상황

### D. 장애 테스트
- PG API 타임아웃
- 콜백 누락
- 오프라인 동기화 지연/부분 실패

## 7) 선행 과제(즉시 수행 권장)
1. 결제 상태 코드 표준화(내부 상태 enum + 외부 PG 코드 매핑표)
2. 주문/결제/환불 식별자 네이밍 통일(orderId, paymentId, pgTxId)
3. 콜백 검증 정책 강제(서명/재전송/순서 역전 처리)
4. 관리자 보정 API 및 감사로그 필수화
5. 운영 환경에서 개발 편의 옵션 비활성화
   - `ALLOW_GENERATED_ADMIN_TOKEN=false`
   - `KICC_USE_MOCK_APPROVE=false`

## 9) 의사결정 원칙
- 상품 계산과 결제를 분리해 결제 안정성을 우선 확보
- 신규 도입(KICC/키오스크)은 기존 운영 안정성(LightPay) 훼손 없이 점진 반영
- 모든 전환은 롤백 가능한 작은 단위로 진행
